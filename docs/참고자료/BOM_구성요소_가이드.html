BOM 구성요소 관계도

WIMS 의 BOM 은 5개 핵심 구성요소로 이루어져 있습니다. 각 박스 위에 마우스를 올려 정의를 확인하세요.

핵심 개념: 표준BOM = EBOM + MBOM + Config 를 하나로 묶은 것입니다. 이 셋은 분리 불가하며, 어느 하나라도 바뀌면 standardBomVersion 이 +1 됩니다.

자재구성 (EBOM)

"이 창호에 어떤 부품이 들어가나?"

  • 기능 단위로 분해: 구조부(프레임), 유리부, 개폐부(철물), 밀봉부, 방충부
  • 설계 관점의 구성 — "무엇이 필요한가"
  • MBOM 과 다대다 매핑

공정구성 (MBOM)

"공장에서 어떤 순서로 만드나?"

  • 공정 단위로 분해: 프레임 절단 → 조립 → 유리 삽입 → …
  • 제조 관점의 구성 — "어떻게 만드는가"
  • 절단 속성(방향·산식·수량)을 직접 보유

옵션구성 (Config)

"고객이 어떤 사양을 선택했나?"

  • 레이아웃(W2XH1-정), 치수(W=1500, H=1200), 유리 종류 등
  • ENUM 옵션(선택형) + NUMERIC 옵션(수치 입력)
  • 상태: DRAFT → RESOLVED → RELEASED

옵션별규칙 (BOM Rule)

"옵션에 따라 BOM을 어떻게 바꾸나?"

  • 조건(when) + 동작(action) 쌍
  • 4가지 동사: SET(값 설정), REPLACE(부품 교체), ADD(행 추가), REMOVE(행 삭제)
  • 산식 포함 가능: W - 94, IIF(H≥900, 2, 0)

확정구성표 (Resolved BOM)

"최종적으로 공장에 내려가는 BOM"

  • Base MBOM + Rule 적용 결과
  • 산식이 실제 수치로 평가된 상태
  • frozen 후 불변 — MES는 이 값을 신뢰

MES (생산시스템)

"확정구성표를 받아 실제로 생산하는 외부 시스템"

  • Resolved BOM을 API로 조회 (GET 전용)
  • 절단 치수·수량 snapshot 값을 그대로 사용
  • 권한: ROLE_MES_READER

📦 제품 등록

모델코드(예: DHS-AE225-D-1)로 제품 마스터를 등록합니다.

제품 분류 4축: 대분류(미서기/커튼월) → 계약구분(마스/우수) → 유리사양(삼중/복층) → 치수크기

파생제품은 기본제품을 참조하고 차이만 Rule 로 표현합니다.

🔧 BOM 구성 (EBOM + MBOM)

EBOM: "어떤 부품이 필요한가" — 기능군(구조부·유리부·개폐부·밀봉부·방충부)별 자재 목록

MBOM: "어떻게 만드는가" — 공정별 자재 + 절단 속성(방향·산식·수량)

EBOM ↔ MBOM 은 EBOM_MBOM_MAP 으로 다대다 매핑. 하나의 EBOM 자재가 여러 MBOM 공정에 쓰일 수 있습니다.

⚙️ 옵션 선택 (DRAFT)

DRAFT 견적 단계에서 고객 요구에 맞는 옵션을 선택합니다.

옵션 예시: 레이아웃(W2XH1-정), 가로 치수(W=1500), 세로 치수(H=1200)

이 단계에서는 Resolved BOM row 를 만들지 않고 계산만 수행합니다 (프리뷰). DRAFT 취소 시 DB 쓰레기가 남지 않습니다.

🔄 Rule 적용 (RESOLVED)

RESOLVED 선택된 옵션 조합에 매칭되는 BOM Rule 을 순서대로 적용합니다.

예: 계열이 225-마스이면 후렘 절단 산식을 W - 94로 SET

산식이 실제 수치로 평가됩니다: W - 94 → W=1500일 때 → 1406mm

결과는 cut_length_evaluated 등 snapshot 필드에 저장됩니다.

✅ 확정 (RELEASED / frozen)

RELEASED 3가지 트리거 중 하나가 발생하면 frozen = TRUE 로 전환됩니다:

  • T1 견적서 CONFIRMED (견적설계 서브시스템)
  • T2 작업지시 RELEASED (제조관리 서브시스템)
  • T3 PM UI "확정" 버튼 (제품관리 서브시스템)

frozen 후에는 산식 상수가 바뀌어도 snapshot 값은 불변. 재평가가 필요하면 DEPRECATED 처리 후 신규 버전을 생성합니다.

🏭 MES 조회 (생산)

MES 시스템이 GET /api/external/v1/bom/resolved/{resolvedBomId} 로 확정구성표를 조회합니다.

응답에 절단 속성(방향·길이·수량의 평가 결과)이 포함되어 바로 작업지시로 활용됩니다.

DEPRECATED 상태가 되면 신규 바인딩은 불가하고 이력 조회만 가능합니다.

버전 · 스냅샷

BOM 의 변경 추적과 확정(frozen) 후 불변성 보장 메커니즘입니다.

📐 standardBomVersion — 통합 버전

EBOM, MBOM, Config 중 어느 하나라도 변경되면 +1 증가합니다.

E
EBOM 변경 → version +1
M
MBOM 변경 → version +1
C
Config 변경 → version +1

changedComponents 필드에 어떤 것이 바뀌었는지 기록됩니다.

🔒 frozen — 확정 후 불변

frozen=TRUE 이후 다음 필드는 절대 변경 불가:

🔒
cut_length_evaluated — 절단 길이 결과
🔒
cut_length_evaluated_2 — 2차 절단 길이
🔒
cut_qty_evaluated — 절단 수량
🔒
actual_qty, actual_cut_length
🔒
rule_engine_version

재평가 필요 시 → DEPRECATED 처리 후 신규 버전 생성

🔑 optionsHash — 옵션 조합 해시

계산 방식:

appliedOptions → ENUM 만 추출 → SHA-256 앞 8자

NUMERIC 옵션(W, H 등)은 해시에서 제외 — 카디널리티 폭발 방지

결과:

W=1500, H=1200 과 W=1501, H=1201 은 같은 resolvedBomId 로 수렴

같은 제품·같은 ENUM 옵션이면 1개 논리 BOM. 실제 치수는 appliedOptions 에 저장, cut_length_evaluated 로 평가·스냅샷.

lazy 생성 정책: DRAFT 단계에서는 RESOLVED_BOM row 를 만들지 않습니다. T1(견적 확정) 또는 T2(작업지시) 트리거 시점에만 INSERT. DRAFT 취소 시 DB 쓰레기가 남지 않습니다.
