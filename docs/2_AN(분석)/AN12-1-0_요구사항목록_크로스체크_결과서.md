## 3. 발견된 문제점

### 3.1 누락 항목 (5건)

| # | 유형 | 내용 | 근거 | 심각도 |
|---|------|------|------|--------|
| N-1 | 기능 누락 | **거래처 CRUD 기능** — 사이트맵에 "거래처관리" 메뉴(원자재 거래처 목록/상세)가 있으나, 거래처 등록·수정·삭제·조회에 대한 별도 기능 요구사항이 없음. FR-PM-003(구매 단가 이력 관리)이 부분 커버하지만 거래처 마스터 관리 자체가 미정의 | AN21 사이트맵 No.43~44 | 상 |
| N-2 | 기능 누락 | **발주 현황 조회 / 재고 현황 조회** — 사이트맵에 발주 현황(/pages/orderstatus), 재고 현황(/:orderNo/stock), 재고 사용(/.../stock) 3개 화면이 있으나 대응 요구사항이 없음 | AN21 사이트맵 No.35, 38, 59 | 중 |
| N-3 | 기능 누락 | **프로젝트 기본 CRUD** — 프로젝트 생성/수정/삭제/상태관리/목록조회는 시스템의 가장 기본적인 기능이나 별도 요구사항이 없음. FR-ES-003(설계자 배정)만 존재 | AN21 사이트맵 No.3~9 | 중 |
| N-4 | 추적성 부족 | **현행 오류 11건 세부 내역 미기재** — FR-CM-005에서 "현행 시스템 오류 수정 (11건 통합)"으로 묶었으나, 11건의 개별 오류가 무엇인지 명시하지 않아 개발 시 범위 파악이 곤란 | AN11-2 오류 11건 | 중 |
| N-5 | 추적성 부족 | **도입 우려사항 5건 → 요구사항 변환 추적이 불명확** — AN12-0에서 5건의 도입 우려사항을 요구사항으로 변환하는 방침을 수립했으나, AN12-1에서 어떤 NFR로 매핑되었는지 명시적 추적이 없음 | AN12-0 §도입우려 | 하 |

### 3.2 중복/과잉 항목 (4건)

| # | 유형 | 내용 | 조치 방안 |
|---|------|------|----------|
| D-1 | 기능-비기능 중복 | **FR-CM-002 (역할 기반 접근제어 RBAC)** vs **NFR-SC-CM-002 (역할 기반 접근제어 정책)** — 기능 요구사항과 비기능 요구사항이 거의 동일한 범위를 다룸 | FR-CM-002는 "RBAC 기능 구현"으로, NFR-SC-CM-002는 "접근제어 정책 수립 및 적용 기준"으로 설명을 명확히 분리 |
| D-2 | 기능-비기능 중복 | **FR-CM-001 (로그인 및 세션 관리)** vs **NFR-SC-CM-001 (JWT 토큰 기반 인증)** — 기능(로그인 화면/동작)과 비기능(인증 방식)으로 분리한 의도는 맞으나 설명 범위가 겹침 | 각 요구사항 설명에서 구분 기준을 명확히 기술 |
| D-3 | 기능 중복 | **FR-ES-018 (실행예산 산출: 별도비/제작비/시공비/수수료)** vs **FR-ES-020 (실행 예산 1차 산출 및 원가율 분석)** — 두 요구사항이 실질적으로 같은 기능의 다른 측면을 기술 | FR-ES-018을 "실행예산 항목별 산출(비용 입력)"로, FR-ES-020을 "실행예산 총괄 분석(원가율/수익성)"으로 범위를 명확히 분리하거나 하나로 통합 |
| D-4 | 비기능 중복 | **NFR-RL-FS-002 (자동 동기화 및 충돌 해결)** vs **NFR-RL-FS-003 (동기화 충돌 해결 규칙)** — 두 요구사항이 "충돌 해결"이라는 동일 주제를 다룸 | NFR-RL-FS-002로 통합하고, "충돌 해결 규칙(타임스탬프 기반)" 내용을 하위 설명으로 흡수 |

### 3.3 비기능 요구사항 수치 불일치 (7건)

AN12-0-4(원본)와 AN12-1(현재 목록)의 성능 기준치가 상이하다. 어느 쪽이 확정 값인지 현업 협의가 필요하다.

| 항목 | AN12-0-4 (원본) | AN12-1 (현재) | 차이 | 판단 |
|------|----------------|--------------|------|------|
| 일반 화면 응답시간 | 3초 이내 | 2초 이내 | 더 엄격 | ⚠ AN12-0-4 원본 기준으로 통일 권장 (2초는 과도할 수 있음) |
| BLE 측정값 수신 지연 | 200ms 이내 | 500ms 이내 | 더 완화 | ⚠ AN12-0-4 기준(200ms)이 BLE 실시간성에 더 적합 |
| 동시접속 기준 | 30명 | 50인 | 더 엄격 | ⚠ 실제 사용자 규모(~6명) 고려하면 30명이면 충분 |
| 최적절단 알고리즘 | 10초 이내 | 30초 이내 | 더 완화 | 30초가 현실적 (NP-hard 계열 알고리즘) |
| MES REST API 응답 | 2초 이내 | 1초 이내 | 더 엄격 | ⚠ API 특성상 1초가 바람직하나, 2초가 현실적 |
| 대용량 도면 | 50MB/10초 | 100MB/안정적 | 기준 상이 | ⚠ 정량적 기준(MB, 초)으로 통일 필요 |
| 반응형 해상도 | 1024px~ | 1280px~ | 더 좁음 | 1280px이 현실적 (업무용 모니터 기준) |

### 3.4 건수 차이 분석

| 구분 | AN12-0-2 예상 | AN12-1 실제 | 차이 | 사유 |
|------|-------------|------------|------|------|
| 기능 요구사항 | 87~116건 | 73건 | -14~43건 | 오류 11건 → 1건 통합, 설문 유사항목 통합, 일부 세부 기능 미분할 |
| 비기능 요구사항 | 49건 | 49건 | 0 | 일치 |
| 합계 | 136~165건 | 122건 | -14~43건 | |

기능 요구사항이 예상 하한(87건)보다도 14건 적다. 통합 자체는 합리적이나, 누락 여부를 재확인할 필요가 있다. → 3.1에서 식별한 누락 3건(거래처 CRUD, 발주/재고 현황, 프로젝트 CRUD)을 추가하면 76건으로 소폭 증가.

## 4. 실현가능성 검토

### 4.1 기술적 리스크

| 요구사항 | 리스크 | 수준 | 비고 |
|---------|--------|------|------|
| FR-ES-001 (CAD 웹 뷰어 변환) | .dwg → 웹 변환 엔진 필요 (Autodesk APS 또는 자체 구현). 현행 시스템에 이미 존재하므로 기존 방식 계승이 현실적 | 중 | 현행 시스템의 Canvas 기반 렌더링 방식 확인 필요 |
| FR-ES-004 (CAD 블록 자동인식) | .dwg 파서에서 블록/외부참조 메타데이터를 추출하는 것은 DWG 포맷 의존적. ODA(Open Design Alliance) SDK 또는 상용 라이브러리 필요 | 상 | 현행 시스템 구현 방식 확인 후 판단 |
| FR-OM-004 (최적 절단 알고리즘) | 1D Cutting Stock Problem — NP-hard 계열. 근사 알고리즘(FFD, BFD) + 메타휴리스틱(SA, GA) 조합 필요 | 중 | 현행 시스템에 이미 있다면 알고리즘 개선 수준 |
| NFR-SC-FS-002 (원격 데이터 삭제) | MDM 솔루션 없이 자체 구현 시 앱이 실행 중이어야 동작하는 제약. 기기 분실 상황에서 실효성 한계 | 중 | 우선순위 "중"으로 적절, 단 구현 범위 축소 검토 |
| FR-FS 전체 (모바일 앱) | Android 앱 전체를 새로 개발 (BLE + Room + 동기화). 모바일 개발자 1인(김현봉)으로 12건을 Phase 2(5개월)에 완료해야 함 | 상 | 모바일 개발 인력 1인 — 일정 리스크 높음 |

### 4.2 일정 리스크

| Phase | 기간 | 기능 요구사항 | 개발 인력 | 1인당 부담 | 판단 |
|-------|------|-------------|----------|-----------|------|
| Phase 1 | 3개월 (M1~M3) | PM 16건 + CM 일부 | BE 2명, FE 2명 | ~5건/인 | ✅ 적정 |
| Phase 2 | 5개월 (M4~M8) | ES 21 + OM 8 + MF 10 + FS 12 = 51건 | BE 2명, FE 2명, 모바일 1명 | ~10건/인 | ⚠ 빡빡함 |

Phase 2에서 특히:
- **ES(21건)**: CAD 뷰어 + 견적 산출 로직이 복잡. FE/BE 모두 투입 필요
- **FS(12건)**: 모바일 개발자 1인이 전담. BLE 연동 + 오프라인 동기화 + UI까지 5개월은 도전적
- **MF(10건)**: 제작도 자동 출력이 핵심 난제. BE 중심
