# WIMS 2.0 요구사항 ID 체계 및 분류 기준

**문서코드:** AN12-0
**버전:** v1.2
**작성일:** 2026.03.25
**작성자:** 김지광
**목적:** 요구사항 정의서(AN12-1) 작성 시 일관된 ID 부여 및 분류를 위한 기준 문서

---

## 1. 요구사항 ID 구성 규칙

### 1.1 ID 형식

```
[분류접두어]-[서브시스템코드]-[일련번호 3자리]
```

**예시:** `FR-PM-001`, `NFR-SC-CM-002`

### 1.2 서브시스템 코드

| 코드 | 서브시스템 | Phase | 비고 |
|------|-----------|-------|------|
| PM | ① 제품관리 시스템 | Phase 1 | BOM/자재/공정/제조룰 |
| ES | ② 견적설계 시스템 | Phase 2 | CAD 뷰어, 견적 산출 |
| OM | ③ 발주관리 시스템 | Phase 2 | 최적절단, 발주서 |
| MF | ④ 제조관리 시스템 | Phase 2 | 제작도, 절단지시서 |
| FS | ⑤ 현장실측 시스템 | Phase 2 | Android 앱, BLE 연동 |
| CM | 공통/시스템 전체 | 전체 | 인증, 권한, UI 공통 등 |

### 1.3 분류 접두어

| 접두어 | 분류 | 설명 |
|--------|------|------|
| FR | 기능 요구사항 | Functional Requirement |
| NFR-PF | 성능 | Performance |
| NFR-SC | 보안 | Security |
| NFR-US | 사용성 | Usability |
| NFR-RL | 신뢰성 | Reliability |
| NFR-MT | 유지보수성 | Maintainability |
| NFR-PT | 이식성 | Portability |
| NFR-DA | 데이터 | Data (이관, 호환성 등) |
| NFR-IF | 인터페이스 | Interface (MES 연동 등) |

### 1.4 일련번호 부여 규칙

- 3자리 숫자, 001부터 시작
- 서브시스템별로 독립 채번 (PM-001, ES-001 각각 시작)
- 삭제된 번호는 재사용하지 않음
- 요구사항 세분화 시: `FR-PM-001-01`, `FR-PM-001-02` (하위 번호 추가)

## 3. 비기능 요구사항 적용 범위

| 적용 범위 | 서브시스템 코드 | 예시 |
|-----------|----------------|------|
| 시스템 전체 공통 | CM | `NFR-SC-CM-001` (JWT 인증) |

## 4. 요구사항 분류(유형) 체계

WIMS 프로젝트는 민간 SI 사업으로 별도의 제안요청서가 없으므로, 특허청 방법론의 요구사항 유형 분류를 아래와 같이 프로젝트에 맞게 적용한다.

### 4.1 분류 유형 정의

| 유형 | 설명 | 해당 접두어 | 출처 예시 |
|------|------|------------|-----------|
| 기능 | 서브시스템별 업무 기능 (CRUD, 화면, 비즈니스 로직) | FR | 개발계획서 §3.2, 설문조사 |
| 성능 | 응답시간, 처리량, 동시접속자 수 등 | NFR-PF | 개발계획서 §3.3.3, 기술 스택 |
| 보안 | 인증, 암호화, 접근제어, 데이터 보호 | NFR-SC | 개발계획서 §3.3.3, 기술 스택 |
| 사용성 | UI/UX, 접근성, 현장 작업 편의, 반응형 대응 | NFR-US | 설문조사, 개발계획서 §3.3.3 |
| 인터페이스 | MES REST API, BLE 연동, 외부 시스템 연계 | NFR-IF | 개발계획서 §3.4, MES 협의 |
| 데이터 | 마이그레이션, 코드체계 표준화, 데이터 정합성 | NFR-DA | 설문조사, 개발계획서 §2.2 |
| 제약사항 | 기술 스택, 호환성(OS/브라우저), 법규 준수 | NFR-PT / NFR-MT | 기술 스택 결정서, 개발계획서 |
| 신뢰성 | 장애 복구, 오프라인 내구성, 백업/복원 | NFR-RL | 개발계획서 §3.3.3 |

### 4.2 출처 기재 기준

특허청 서식의 "출처" 항목에는 다음과 같이 기재한다.

| 출처 유형 | 기재 방법 | 예시 |
|-----------|-----------|------|
| 설문조사 | 사전설문조사 결과서(AN11-2) + 영역·번호 | AN11-2, 견적영역-Q3 |
| 개발계획서 | 개발계획서 + 절·항 번호 | 개발계획서 v1.2, §3.2 |
| 업체 요청 | 업체요청 + 요청자명 | 업체요청(유미숙) |
| BOM 부록 | BOM 구조 정의서(DE35-1) + 부록 번호 | DE35-1, 부록A |
| MES 협의 | MES 연동 협의 + 일자 | MES 협의(2026.03) |
| 자체 도출 | 수행업체 분석 | 수행업체 분석 |

## 5. 난이도 판단 기준

| 난이도 | 판단 기준 | 대표 사례 |
|--------|-----------|-----------|
| **상** | 신규 알고리즘 개발, 외부 시스템 연동 프로토콜 설계, 복잡한 비즈니스 룰 구현, 기존에 레퍼런스가 없는 기능 | 최적 절단 알고리즘, BLE GATT 프로토콜 연동, BOM 다단계 트리 버전 관리, MES REST API 설계 |
| **중** | 표준 CRUD + 비즈니스 로직, 기존 패턴 응용, 화면-서버 간 연동, 데이터 가공/변환 | BOM 계층 등록·조회, 견적서 자동 산출, 개소별 치수 매핑 UI, 오프라인 동기화 |
| **하** | 단순 화면 개선, 메타정보 관리, 코드 정비, 기본 CRUD, 목록 조회 | 거래처 목록 UI 개선, 제품 분류 체계 관리, 자재 코드 속성 확장, 계약 이력 조회 |

### 난이도 판단 시 고려 요소

- **기술 복잡도** — 새로운 기술/프로토콜 도입 여부
- **비즈니스 복잡도** — 도메인 규칙의 복잡성 및 예외 케이스 수
- **연동 범위** — 외부 시스템 또는 타 서브시스템과의 의존성
- **구현 불확실성** — 요구사항이 명확한지, 추가 분석이 필요한지

## 6. 수용여부 판단 기준

특허청 방법론 서식(AN12-1)에 따라 각 요구사항에 수용/일부수용/미수용을 판정한다.

### 6.1 판정 기준

| 판정 | 기준 | 요구사항 정의서 기재 방법 |
|------|------|-------------------------|
| **수용** | 개발 범위(8개월, 2-Phase) 내에서 구현 가능하고, 기술적 제약이 없는 요구사항 | 수용여부: ☑ 수용 |
| **일부수용** | 핵심 기능은 구현하되, 범위·일정·기술적 이유로 일부 축소 또는 대안 제공 | 수용여부: ☑ 일부수용 + 사유·대안 기재 |
| **미수용** | 현재 프로젝트 범위 외, 기술적 불가, 별도 사업 필요, 또는 투입 대비 효과 미미 | 수용여부: ☑ 미수용 + 미수용 사유 상세 기재 |

### 6.2 일부수용 적용 기준 (상세)

| 일부수용 사유 유형 | 설명 | 예시 |
|-------------------|------|------|
| **범위 축소** | 요구사항의 일부 기능만 현재 Phase에서 구현 | "MES 연동 요청 → Phase 1에서 BOM 데이터 전송 우선, 나머지 향후 확장" |
| **대안 제공** | 요청한 방식 대신 기술적으로 더 적합한 대안 제시 | "실시간 알림 요청 → 대시보드 새로고침 방식으로 대체" |
| **일정 이연** | Phase 2 이후 또는 안정화 기간에 구현 | "모바일 견적 조회 → Phase 2 후반부 안정화 시점 구현" |
| **기능 간소화** | 자동화 범위를 축소하여 반자동 또는 수동 보조로 구현 | "AI 자동 제품 판별 → 규칙 기반 추천 + 수동 확정" |

### 6.3 미수용 사유 유형

| 미수용 사유 유형 | 설명 | 예시 |
|-----------------|------|------|
| **범위 외** | 현재 프로젝트 범위에 포함되지 않는 요구사항 | "회계 시스템 연동", "인사 관리 기능" |
| **기술적 불가** | 현재 기술 스택이나 환경에서 구현 불가 | "iOS 앱 동시 개발" (Android만 범위) |
| **별도 사업** | 규모·성격상 독립 프로젝트로 추진해야 함 | "전사 MES 고도화", "AI 기반 자동 설계" |
| **비용 대비 효과 미미** | 구현 공수 대비 사용 빈도·효과가 낮음 | 극히 특수한 1회성 기능 요청 |

### 6.4 도입 우려사항(5건) 처리 방침

설문조사의 "도입 우려사항" 5건은 기능 요구사항이 아닌 리스크/우려 항목이므로, 요구사항 정의서에 직접 포함하지 않고 아래와 같이 변환한다.

| 우려사항 | 변환 대상 | 변환 후 분류 | 요구사항 ID 체계 |
|---------|----------|-------------|-----------------|
| 기존 데이터 이관 안전성 | 비기능 요구사항 — 데이터 | NFR-DA-CM | 데이터 이관 검증 절차 요건 |
| 병행 운영 기간 필요 | 제약사항 | NFR-PT-CM | 병행 운영 기간 확보 요건 |
| UI/UX 이질감 최소화 | 비기능 요구사항 — 사용성 | NFR-US-CM | 현행 UI 패턴 계승 요건 |
| 충분한 교육·매뉴얼 | 제약사항 | NFR-US-CM | 사용자 교육·매뉴얼 제공 요건 |
| 요청사항 완수 우려 | 프로젝트 관리 (RTM 관리) | — | 요구사항 추적표(AN14-1)에서 관리 |
