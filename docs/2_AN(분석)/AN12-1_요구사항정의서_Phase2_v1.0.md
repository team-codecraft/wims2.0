## 1. 개요

### 1.1 목적

본 문서는 WIMS 2.0 Phase 2 (M4~M8) 구현 범위의 기능 요구사항(FR) 53건과 비기능 요구사항(NFR)을 특허청 SW 개발방법론 AN12-1 형식으로 정의한다. 분석 단계에서 도출된 요구사항을 설계·구현 단계에서 활용 가능하도록 구체화한다.

### 1.2 범위

**대상 서브시스템 (4개):**
- ② 견적설계 시스템 (Estimation & Design, ES)
- ③ 발주관리 시스템 (Order Management, OM)
- ④ 제조관리 시스템 (Manufacturing, MF)
- ⑤ 현장실측 시스템 (Field Survey, FS)

**적용 기간:** Phase 2 (M4~M8) = 스프린트 S6~S11 (약 24주)

**제외 범위:**
- Phase 1 (① 제품관리 시스템) — 별도 AN12-1-P1 문서 참조
- Phase 3 이후 요구사항

### 1.4 용어 정의

| 용어 | 정의 |
|------|------|
| **개소** | 도면 상의 창호 설치 위치. 건물 평면도에서 A-1, B-2 등으로 표시되는 각 창호/도어 설치점 |
| **CAD 도면** | AutoCAD .dwg, .dxf 형식의 건축 설계도. 개소 정보를 포함한 2D/3D 도면 |
| **BOM (Bill of Materials)** | 제조BOM. 각 제품 규격별 필요한 원자재(프로필, 유리), 부자재(하드웨어), 공정비, 제조룰(가공비, 조립비) 포함 |
| **롤방충망** | 창호에 설치되는 방충망. 롤 형태로 보관하며, 규격(높이×폭)별 중량을 산출하는 기능 |
| **미서기** | 미닫이 창호의 일종. 보강재(하드웨어) 적용 규칙이 있어 자동화 로직 필요 |
| **트랜섬** | 창호의 가로 프레임(수평재). 절단 사이즈가 다른 부재와 연관 |
| **실행예산** | 제조사 내부용 예산. 원가율, 수익성 검토용으로 자재비·시공비·기타비용 합산 |
| **절단 최적화** | 원자재 낭비를 최소화하는 절단 조합 선택. NP-hard 계열 알고리즘 |

## 3. 기능 요구사항 상세 (53건)

### 3.1 견적설계 시스템 (ES, 21건)

#### FR-ES-001 CAD 도면 업로드 및 웹 뷰어 변환

| 항목 | 내용 |
|------|------|
| **요구사항 ID** | [[AN12-1_요구사항목록_v1.5#1.2 ② 견적설계 시스템 (ES) — 21건\|FR-ES-001]] |
| **요구사항명** | CAD 도면 업로드 및 웹 뷰어 변환 |
| **분류** | 기능 > 견적설계 > 도면관리 |
| **난이도** | 상 |
| **수용여부** | 수용 |
| **우선순위** | 최상 |
| **출처** | AN11-2(설문), 개발계획서 §3.2② |
| **관련 요구사항** | FR-ES-002, FR-ES-004, NFR-PF-ES-001 |
| **현행 화면 매핑** | 도면 뷰어 > 도면 선택기 |

**상세 설명:**

사용자가 클라이언트(발주처) 제공 CAD 도면(.dwg, .dxf)을 웹 브라우저에서 업로드하면, 서버가 변환 서비스를 통해 웹 뷰어(SVG 또는 WebGL)로 렌더링할 수 있는 형식으로 변환한다. 변환 완료 후 브라우저에서 줌, 팬, 레이어 표시/숨김, 측정 등 기본 CAD 뷰어 기능을 제공한다.

**입력 데이터:**
- CAD 파일: .dwg, .dxf (최대 50MB)
- 파일 메타정보: 버전, 좌표계, 단위(mm/cm/m)

**출력 데이터:**
- 변환된 도면 바이너리 (SVG 또는 WebGL 포맷)
- 도면 썸네일 이미지
- 레이어 목록 메타정보

**비즈니스 규칙:**
- 변환 타임아웃: 5분 초과 시 실패 처리
- 지원 CAD 버전: AutoCAD 2010 이상
- 변환 실패 시 업로드 재시도 UI 제공
- 변환된 도면은 프로젝트 스토리지에 30일 보관

**수용 사유:**
견적설계의 출발점. 현행 Navisworks 기반 뷰어를 웹 기반으로 전환하여 설치/유지보수 비용 절감, 태블릿 웹 접근성 향상.

**구현 시 고려사항:**
- 외부 라이브러리: Three.js, Babylon.js 또는 CAD 변환 서비스(예: CloudConvert, Aspose)
- BE: Spring MVC 컨트롤러 + 파일 스토리지(S3), 비동기 변환 큐(SQS)
- FE: Vue 3 캔버스 컴포넌트, 반응형 줌/팬 제스처 처리

#### FR-ES-002 도면 내 개소 라벨링 및 관리

| 항목 | 내용 |
|------|------|
| **요구사항 ID** | [[AN12-1_요구사항목록_v1.5#1.2 ② 견적설계 시스템 (ES) — 21건\|FR-ES-002]] |
| **요구사항명** | 도면 내 개소 라벨링 및 관리 |
| **분류** | 기능 > 견적설계 > 도면관리 |
| **난이도** | 중 |
| **수용여부** | 수용 |
| **우선순위** | 상 |
| **출처** | AN11-2(설문), 개발계획서 §3.2② |
| **관련 요구사항** | FR-ES-001, FR-ES-003, FR-ES-005 |
| **현행 화면 매핑** | 도면 뷰어 > 개소 추가/수정 |

**상세 설명:**

도면 렌더링 후 사용자가 도면 상의 창호/도어 설치 위치(개소)에 라벨(A-1, A-2, B-1 등)을 마킹한다. 각 개소에 대해 다음 정보를 관리:
- 라벨 텍스트 및 좌표 (도면 상 픽셀 또는 실제 좌표)
- 개소 타입 (창호/도어/프레임/기타)
- 높이, 폭 등 초기 치수 정보

**입력 데이터:**
- 도면 이미지 좌표 (클릭/드래그)
- 라벨명: 영문+숫자 조합 (예: A-1, Window-01)
- 개소 타입 선택
- 선택적 좌표: 도면의 실제 건축 좌표(mm 단위)

**출력 데이터:**
- 라벨 마크업 JSON (각 개소의 좌표, 라벨, 타입, 치수)
- 개소 목록 표 (도면과 연동)

**비즈니스 규칙:**
- 라벨은 고유해야 함 (중복 금지)
- 라벨 명명 규칙: 영문 1~3자 + 하이픈 + 숫자 (예: A-1, WALL-05)
- 개소 삭제 시 해당 견적 데이터 자동 삭제 여부 확인 후 진행
- 도면 재로드 시 기존 라벨 정보 자동 복원

**수용 사유:**
견적설계의 필수 단계. 수작업으로 도면에 개소를 표시하는 현행 방식을 디지털화.

#### FR-ES-003 설계자 배정 및 진행 현황 관리

| 항목 | 내용 |
|------|------|
| **요구사항 ID** | [[AN12-1_요구사항목록_v1.5#1.2 ② 견적설계 시스템 (ES) — 21건\|FR-ES-003]] |
| **요구사항명** | 설계자 배정 및 진행 현황 관리 |
| **분류** | 기능 > 견적설계 > 프로젝트 |
| **난이도** | 중 |
| **수용여부** | 수용 |
| **우선순위** | 상 |
| **출처** | AN11-2(설문) |
| **관련 요구사항** | FR-ES-005, FR-OM-001 |
| **현행 화면 매핑** | 견적 관리 > 설계 할당 |

**상세 설명:**

프로젝트별로 담당 설계자(BOM 매핑, 견적 산출 담당)를 배정하고, 각 프로젝트의 진행 단계를 추적한다:
- 신규 입력 → 개소 라벨링 → 도면 매핑 → BOM 입력 → 견적 검토 → 최종 확정

**입력 데이터:**
- 프로젝트 ID
- 설계자 사용자 ID
- 진행 상태 (선택: 신규, 설계중, 검토, 확정)

**출력 데이터:**
- 설계자별 할당 프로젝트 목록 (진행률 표시)
- 프로젝트별 진행 단계 히스토리
- 타임라인 차트 (Gantt)

**비즈니스 규칙:**
- 한 프로젝트는 하나의 설계자만 할당 가능 (또는 다중 할당 시 역할 구분)
- 상태 전이: 신규 → 설계중 → 검토 → 확정 (역순 이동 가능, 관리자만)

**수용 사유:**

- 중복 인식 방지: 겹침 거리 5mm 이상 떨어진 객체만 별도 개소로 분류

**구현 시 고려사항:**
- CAD 파일 파싱: Open Design Alliance (ODA) 라이브러리 또는 LibreCAD
- BE: 블록 인식 알고리즘 (좌표 매칭, 회전/스케일 보정)
- FE: 제안 목록 UI, 클릭으로 개소 일괄 추가/필터링 기능

- 최적 절단 조합: 표준 롤 길이(10m 등) 내에서 낭비 최소 조합 추천

전원 참여 요청 사항. 현행 수작업 산출 → 자동화로 시간 단축, 오류 감소. 매출 원가율 개선.

**구현 시 고려사항:**
- BE: 최적 절단 알고리즘 (NP-hard, 동적계획법 또는 그리디 근사)
- 마스터 데이터: 방충망 규격, 롤 높이, 단가 테이블
- FE: 산출 결과 시뮬레이션 차트 (높이별 롤 개수, 낭비 시각화)

#### FR-ES-016 자재비·시공비·기타비용 합산 및 견적서 산출

|------|------|

| **난이도** | 중 |
| **수용여부** | 수용 |
| **우선순위** | 최상 |
| **출처** | AN11-2(설문), 개발계획서 §3.2② |
| **관련 요구사항** | FR-ES-007, FR-ES-011, FR-ES-017, FR-ES-019 |
| **현행 화면 매핑** | 견적서 > 작성 |

**상세 설명:**

개소별로 산출된 자재비, 시공비(노동비), 기타비용(수송비, 설치비 등)을 합산하여 최종 견적 금액을 계산하고, 견적서 문서를 생성한다.

**비용 구성:**
- 자재비 = Σ(BOM 자재 단가 × 수량)
- 시공비 = 자재비 × 시공비율(%) [설정값]
- 기타비용 = 자재비 × 기타비율(%) [설정값]
- **소계** = 자재비 + 시공비 + 기타비용
- **부가세(VAT)** = 소계 × 10%
- **총 견적액** = 소계 + 부가세

**입력 데이터:**
- 개소별 BOM (자재, 수량, 단가)
- 시공비율, 기타비율 (프로젝트/회사 기본값)
- 할인율 (선택사항)

**출력 데이터:**
- 견적서 계산 시트:
  - 개소별 항목 상세
  - 자재비/시공비/기타비용 구분
  - 소계, 부가세, 총액
- 견적서 PDF 문서 (FR-ES-019)

**비즈니스 규칙:**
- 부가세 계산: 소계 기준 10% (설정 변경 가능)
- 할인: 총액의 일괄 할인 또는 항목별 할인
- 반올림: 원 단위 (일반적으로 5원 단위)
- 견적 유효기간: 발주 후 3개월 [설정값]

**수용 사유:**
견적서 작성의 핵심. 정확한 가격 산출로 신뢰성 향상.

견적 기준 제조원가율 대비 실제 제조에 소요된 원가율(실행율)을 정확히 계산하고, 수익성을 분석한다. 현행 수작업 계산에서 오류가 많았던 부분을 자동화.

**실행율 계산:**
- 원가 구성: 자재비(주요) + 시공비(부)
- 견적 원가율(%) = 견적 원가 / 견적 총액 × 100
- 실행 원가율(%) = 실제 원가(실행예산) / 실제 매출액 × 100
- **실행율(%) = 실행 원가율 / 견적 원가율 × 100**
  - 100% 이상: 손실 (과원가)
  - 100% 미만: 이익

**입력 데이터:**
- 견적 원가 정보 (FR-ES-016)
- 실행예산 원가 정보 (FR-ES-018)
- 매출액

**출력 데이터:**
- 원가율 (%), 실행율 (%)
- 이익/손실액
- 원가 변동 분석 (자재비 증감, 시공비 변동 등)
- 추이 차트

**비즈니스 규칙:**
- 계산 기준: 소재비(VAT 제외) 기준
- 반올림: 소수점 2자리
- 분석 대상: 완공된 프로젝트만

**수용 사유:**
경영진 의사결정의 핵심 지표. 현행 수작업 오류 → 자동 계산으로 신뢰도 향상.

**상세 설명:**

**예시:**
- 필요: 프로필 1200mm × 15개, 1500mm × 12개, 800mm × 8개
- 표준 정척: 6000mm
- 최적 조합: 6000mm 정척 7개 사용 (절단 방법을 자동 결정)
- 결과 폐기율: 약 5% (수작업 대비 20%)

**입력 데이터:**
- 필요 자재 목록 (규격별 수량)
- 표준 정척 마스터 (자재별 표준 길이/면적)
- 절단 공차 허용값

**출력 데이터:**
- 최적 절단 조합:
  - 정척 개수, 절단 패턴 (각 정척에서 뭘 어떻게 자를지)
  - 폐기율 (%)
  - 소요 비용 추정
- 거래처별 발주서 생성 준비

**비즈니스 규칙:**
- NP-hard 문제 (조합 최적화), 동적계획법 또는 휴리스틱 알고리즘 사용
- 다양한 자재(프로필, 유리, 부자재)별로 독립적으로 계산
- 공차: 자재별로 다름 (예: 프로필 ±5mm, 유리 ±2mm)
- 최적화 우선순위: 비용 절감 > 납기 단축

**수용 사유:**
원가 절감의 최대 효과. 수작업 대비 폐기율 70% 감소 가능. 매출 순익률 향상의 가장 큰 기회.

**구현 시 고려사항:**
- 복잡한 알고리즘: 전문 라이브러리(예: OptaPlanner) 고려
- 계산 시간: NFR-PF-OM-001 참조 (1,000개 자재 기준 10초 이내)

```
POST /api/v1/field-survey/measurements
Content-Type: application/json
Authorization: Bearer {JWT_TOKEN}

{
  "project_id": "PRJ001",
  "area_id": "A-1",

    {
      "type": "height",

      "memo": "문틀 손상"
    }
  ]
}
```

**응답:**
- 성공(200): {status: "success", id: "measurement_id"}
- 검증 오류(400): {status: "error", message: "유효성 검증 실패", errors: [...]}
- 인증 오류(401): {status: "error", message: "JWT 토큰 만료"}

- 재시도: 전송 실패 시 지수 백오프로 최대 5회 재시도
- JWT 인증: Authorization 헤더에 토큰 포함
- 데이터 압축: gzip 압축으로 데이터 전송량 최소화

**구현 시 고려사항:**
- 라이브러리: Retrofit2 + OkHttp3 (Android HTTP 클라이언트)
- 인터셉터: JWT 자동 갱신, 요청/응답 로깅
- 예외 처리: 네트워크 타임아웃, SSL 오류 등 명확한 오류 메시지

- Room DB 암호화 설정
- SQLCipher 또는 Jetpack Security 라이브러리 통합
- 앱 시작 시 자동 복호화

**검증 방법:**
- 데이터베이스 파일 검사 (암호화 확인)

| **관련 기능 요구사항** | FR-FS-010 |

**상세 기술:**

측정 데이터를 서버로 전송할 때 네트워크 오류나 서버 오류(5xx)가 발생하면 자동으로 재시도하여 최대한 전송 성공률을 높여야 한다.

**재시도 정책:**
- 재시도 알고리즘: 지수 백오프 (Exponential Backoff)
- 초기 대기 시간: 1초
- 최대 재시도 횟수: 5회
- 백오프 계산: 2^n초 (1, 2, 4, 8, 16초 → 최대 31초)
- 전체 재시도 소요 시간: ~60초

**재시도 대상 오류:**
- 네트워크 타임아웃
- 500/502/503 서버 오류
- 일시적 연결 오류

**재시도 제외 오류:**
- 400/401/403/404 클라이언트 오류 (즉시 실패)
- 데이터 유효성 검사 실패

**사용자 알림:**
- 재시도 진행 중: 로딩 인디케이터 표시
- 최종 실패 시: 에러 메시지 및 수동 재시도 옵션 제공

**검증 방법:**
- 네트워크 지연 시뮬레이션 (throttle)
- 서버 오류 주입 (mock 5xx 응답)

| **관련 기능 요구사항** | FR-FS-001, FR-FS-002 |

**상세 기술:**

현장실측 모바일 앱과 BLE 측정기 간 데이터 통신을 위한 BLE GATT 서비스 및 특성(Characteristic)의 규격을 명확하게 정의해야 한다. 양쪽 팀(모바일/하드웨어)이 독립적으로 개발할 수 있도록 상세한 인터페이스 정의가 필수이다.

**GATT 구조:**
```
BLE Service (UUID: {service_uuid})
├─ Characteristic (측정값 데이터)
│  ├─ UUID: {char_uuid}
│  ├─ Properties: Read, Notify
│  ├─ Format: 구조체 또는 바이너리 (정의 필요)
│  └─ 예: [X좌표(4B), Y좌표(4B), Z좌표(4B), 타임스탐프(4B)]
├─ Characteristic (제어 명령)
│  ├─ UUID: {cmd_uuid}
│  ├─ Properties: Write
│  └─ 예: [CMD_START, CMD_STOP, CMD_CALIBRATE]
└─ Characteristic (배터리 상태)
   ├─ UUID: {batt_uuid}
   ├─ Properties: Read, Notify
   └─ Format: 0~100 (%)
```

**정의 항목:**
- 모든 UUID 할당
- 데이터 포맷 및 바이트 순서 (big-endian/little-endian)
- 측정값 범위 및 정밀도 (±0.1mm 등)
- 알림(Notification) 주기 및 조건
- 에러 코드 및 상태 응답

**문서화:**
- 별도 GATT 정의서 (DE24-1 인터페이스 설계서 부록으로 작성)
- 하드웨어 팀과 협의 후 확정
- GATT Database Tool(Android Studio)로 검증

**검증 방법:**
- GATT 명세 대로 구현 확인
- nRF Connect 또는 BluetoothLE Explorer로 GATT 구조 검증
- 실제 기기로 읽기/쓰기/알림 동작 확인
