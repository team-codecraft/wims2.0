# FS 현장실측 요구사항

> [!abstract]
> 현장실측(Field Survey) 서브시스템 기능 요구사항 12건 + 관련 NFR 13건. Android 모바일 앱(BLE 레이저 측정기 연동) 기반 현장 실측·검증·동기화 및 WIMS 웹 대시보드 연동을 포함한다. 모바일 전용 NFR (보안·성능·사용성·신뢰성·인터페이스·이식성) 이 밀집 배치된다.

## 1. FS 서브시스템 개요

| 항목 | 내용 |
|------|------|
| 서브시스템 | ⑤ 현장실측 (Field Survey, FS) |
| 스프린트 | S10~S11 (M7~M8) + 웹 연동 S11 |
| FR 건수 | 12건 |
| NFR 건수 | 13건 (SC 2, PF 3, US 2, RL 4, IF 2, PT 2 — 모두 FS 전용) |
| 플랫폼 | Android (minSdk 29 = Android 10), BLE 4.2+ |
| 핵심 가치 | BLE 측정기 페어링·오프라인 대응·검증 워크플로우로 현장 생산성 향상 |
| 주요 사용자 | 현장 실측 담당자, WIMS 웹 검증 관리자 |
| 주요 접점 | FS 앱 ↔ WIMS 서버 REST API → MF(FR-MF-001 실측 수신) |

**주요 기능 영역:**

1. BLE연동 (FR-FS-001, 009): 측정기 페어링·배터리/연결 상태
2. 데이터입력 (FR-FS-002): BLE 자동 + 수동 입력·유효성 검증
3. 현황관리·대시보드 (FR-FS-003, 007): 앱 내 + 웹 연동 대시보드
4. 오프라인·동기화 (FR-FS-004, 005, 006): 로컬 저장·서버 동기화·프로젝트 캐싱
5. 품질관리 (FR-FS-008): 오류 알림·재측정 요청
6. 문서·보고서 (FR-FS-010, 011, 012): 사진·메모·CSV/PDF 내보내기

## 2. 기능 요구사항 상세 (12건)

#### FR-FS-001 BLE 레이저 측정 장치 페어링 및 연결

| 항목 | 내용 |
|------|------|
| **요구사항 ID** | [[AN12-1_요구사항목록_v1.5#1.5 ⑤ 현장실측 시스템 (FS) — 12건\|FR-FS-001]] |
| **요구사항명** | BLE 레이저 측정 장치 페어링 및 연결 |
| **분류** | 기능 > 현장실측 > BLE연동 |
| **난이도** | 중 |
| **수용여부** | 수용 |
| **우선순위** | 최상 |
| **출처** | 개발계획서 §3.3.1 |
| **관련 요구사항** | FR-FS-002, NFR-RL-FS-004 |

**상세 설명:**

안드로이드 앱에서 BLE(Bluetooth Low Energy 4.2+) 레이저 측정 장치를 자동으로 탐색하고 페어링한다. 이미 페어링된 장치는 자동 재연결되며, 연결 상태를 실시간으로 표시한다.

**기능 요구사항:**
- BLE 장치 탐색: 근처의 BLE 측정기 스캔 (신호 강도 30dBm 이상)
- 자동 페어링: 선택한 장치와 Bluetooth 페어링 자동 수행
- 재연결: 앱 재시작 시 이전 연결 장치 자동 재연결
- 상태 표시: 연결 상태 아이콘 및 배터리 레벨 표시
- 다중 장치: 향후 복수 측정기 동시 연결 지원 구조 설계

**구현 시 고려사항:**
- Android BLE API (android.bluetooth.*)
- 권한 요청: BLUETOOTH, BLUETOOTH_ADMIN, ACCESS_FINE_LOCATION, **BLUETOOTH_SCAN**, **BLUETOOTH_CONNECT** (Android 12 / API 31+ 필수)
- 배터리 최적화: 스캔 간격 조정, 연결 해제 시 스캔 중지
- 예외 처리: 장치 오프라인, 신호 끊김 시 재연결 로직

#### FR-FS-002 실측 데이터 입력 및 유효성 검증

| 항목 | 내용 |
|------|------|
| **요구사항 ID** | [[AN12-1_요구사항목록_v1.5#1.5 ⑤ 현장실측 시스템 (FS) — 12건\|FR-FS-002]] |
| **요구사항명** | 실측 데이터 입력 및 유효성 검증 |
| **분류** | 기능 > 현장실측 > 데이터입력 |
| **난이도** | 중 |
| **수용여부** | 수용 |
| **우선순위** | 최상 |
| **출처** | 개발계획서 §3.3.2 |
| **관련 요구사항** | FR-FS-001, FR-MF-001 |

**상세 설명:**

BLE 측정 장치로부터 수신한 측정값(거리, 면적, 체적) 및 수동 입력값을 개소별로 정리하고, 즉시 유효성 검증한다. 허용 범위를 벗어난 값은 경고 표시하고 재측정을 요청한다.

**기능 요구사항:**
- BLE 자동 입력: 측정기에서 수신한 값이 자동으로 입력 필드에 반영
- 수동 입력: 키보드 또는 음성 입력 지원
- 현장 1차 검증 — 도면 기준값 ±10% (경미한 즉시 피드백용, 사용자 재측정 권고)
- 경고 표시: 범위 벗어나면 주황색 경고, 재측정 권고
- 측정 이력: 동일 항목 재측정 시 이전 값 보관
- 사진 첨부: 각 개소별 측정 현장 사진 최대 5장 첨부
- 현장 메모: 특이사항 기록 (예: "문틀 손상", "높이 불균일" 등)

**비즈니스 규칙:**
- 유효 범위: 도면 기준값 ±10% (추후 조정 가능)
- 필수 항목: 높이, 폭 (최소 2개 항목 필수)
- 면적 검증: (높이 × 폭) 자동 계산 후 오류 시 경고
- 본 FR 은 2단계 검증 체계의 1차(현장·모바일) 검증 — 서버측 2차 엄격 검증은 [[AN12-1_요구사항정의서_Phase2/sections/MF_제조관리#FR-MF-002 수신 데이터 검증 및 이상값 알림|FR-MF-002]] 참조

#### FR-FS-003 개소별 측정 진행 현황 관리

| 항목 | 내용 |
|------|------|
| **요구사항 ID** | [[AN12-1_요구사항목록_v1.5#1.5 ⑤ 현장실측 시스템 (FS) — 12건\|FR-FS-003]] |
| **요구사항명** | 개소별 측정 진행 현황 관리 |
| **분류** | 기능 > 현장실측 > 현황관리 |
| **난이도** | 하 |
| **수용여부** | 수용 |
| **우선순위** | 상 |
| **출처** | 개발계획서 §3.3.2 |
| **관련 요구사항** | FR-FS-002 |

**상세 설명:**

프로젝트 내의 모든 개소를 목록으로 표시하고, 각 개소의 측정 진행 상태를 시각화한다. 미측정, 측정중, 검증완료 등 3단계로 구분한다.

**상태 정의:**
- **미측정**: 아직 측정하지 않은 상태 (회색)

**기능:**
- 진행률 표시: 프로젝트별, 동 별 진행률 바 표시
- 필터링: 상태별, 동 별 필터
- 정렬: 개소명, 진행 상태, 측정 시간순 정렬

- 동기화 상태: 대기 중, 동기화 중, 완료 상태 표시
- 로컬 삭제: 동기화 완료 후 로컬 데이터 삭제 옵션

**데이터 구조:**
- 테이블: measurement_local (ID, project_id, area_id, type, value, timestamp, photo_paths, memo)
- 인덱스: project_id, area_id, timestamp
- 갱신 시간: 측정 직후 로컬 DB에 INSERT, 동기화 시 서버로 전송

#### FR-FS-005 WIMS 서버와 실측 데이터 동기화 (REST API)

| 항목 | 내용 |
|------|------|
| **요구사항 ID** | [[AN12-1_요구사항목록_v1.5#1.5 ⑤ 현장실측 시스템 (FS) — 12건\|FR-FS-005]] |
| **요구사항명** | WIMS 서버와 실측 데이터 동기화 (REST API) |
| **분류** | 기능 > 현장실측 > 서버연동 |
| **난이도** | 상 |
| **수용여부** | 수용 |
| **우선순위** | 최상 |
| **출처** | 개발계획서 §3.3.2, §4.2 P2-3 |
| **관련 요구사항** | FR-FS-004, FR-MF-001, NFR-IF-FS-002, NFR-RL-FS-002, NFR-RL-FS-003 |

**상세 설명:**

안드로이드 앱에서 수집한 실측 데이터를 WIMS 백엔드 REST API를 통해 전송한다. 서버는 데이터를 검증하고 저장한 후 응답한다.

**REST API 엔드포인트:**
```
POST /api/v1/field-survey/measurements
Content-Type: application/json
Authorization: Bearer {JWT_TOKEN}

{
  "project_id": "PRJ001",
  "area_id": "A-1",
  "measurements": [
    {
      "type": "height",
      "value": 2850.5,
      "unit": "mm",
      "timestamp": "2026-04-10T14:30:00Z",
      "photo_paths": ["photo_a1_001.jpg"],
      "memo": "문틀 손상"
    }
  ]
}
```

**응답:**
- 성공(200): {status: "success", id: "measurement_id"}
- 검증 오류(400): {status: "error", message: "유효성 검증 실패", errors: [...]}
- 인증 오류(401): {status: "error", message: "JWT 토큰 만료"}

**기능 요구사항:**
- 배치 전송: 여러 개소의 데이터를 한 번에 전송
- 응답 시간: 정상 네트워크 환경에서 단건 2초 이내 (NFR-PF-FS-002), 오프라인→온라인 복구 시 일괄 동기화 30초 이내 (NFR-PF-FS-003)
- 재시도: 전송 실패 시 지수 백오프로 최대 5회 재시도
- JWT 인증: Authorization 헤더에 토큰 포함
- 데이터 압축: gzip 압축으로 데이터 전송량 최소화

**구현 시 고려사항:**
- 라이브러리: Retrofit2 + OkHttp3 (Android HTTP 클라이언트)
- 인터셉터: JWT 자동 갱신, 요청/응답 로깅
- 예외 처리: 네트워크 타임아웃, SSL 오류 등 명확한 오류 메시지

**재시도 제외 오류:**
- 400/401/403/404 클라이언트 오류 (즉시 실패)
- 데이터 유효성 검사 실패

**사용자 알림:**
- 재시도 진행 중: 로딩 인디케이터 표시
- 최종 실패 시: 에러 메시지 및 수동 재시도 옵션 제공

**검증 방법:**
- 네트워크 지연 시뮬레이션 (throttle)
- 서버 오류 주입 (mock 5xx 응답)
- 재시도 로그 확인

```
BLE Service (UUID: {service_uuid})
├─ Characteristic (측정값 데이터)
│  ├─ UUID: {char_uuid}
│  ├─ Properties: Read, Notify
│  ├─ Format: 구조체 또는 바이너리 (정의 필요)

│  ├─ UUID: {cmd_uuid}
│  ├─ Properties: Write
│  └─ 예: [CMD_START, CMD_STOP, CMD_CALIBRATE]
└─ Characteristic (배터리 상태)
   ├─ UUID: {batt_uuid}
   ├─ Properties: Read, Notify
   └─ Format: 0~100 (%)
```

**정의 항목:**
- 모든 UUID 할당
- 데이터 포맷 및 바이트 순서 (big-endian/little-endian)

**검증 방법:**

- 에뮬레이터 + 실제 기기 모두 테스트

**검증 방법:**
- Gradle: `minSdkVersion 29` 설정 확인
- Android Lint: API 호환성 경고 점검
- 다중 버전 기기 테스트

#### NFR-PT-FS-002 BLE 4.2+ 장치 호환성

| 항목 | 내용 |
|------|------|
| **요구사항 ID** | [[AN12-1_요구사항목록_v1.5#2.7 이식성 (NFR-PT) — 4건\|NFR-PT-FS-002]] |
| **난이도** | 하 |
| **우선순위** | 상 |
| **출처** | AN12-0-4 비기능 보완, 개발계획서 §3.3.3 |
| **관련 기능 요구사항** | FR-FS-001, FR-FS-002 |

**상세 기술:**

현장실측 모바일 앱은 BLE 4.2 이상의 칩셋을 탑재한 모든 Android 기기에서 BLE 측정기와 통신할 수 있어야 한다.

**호환성 범위:**
- BLE 4.2+ (Bluetooth Low Energy)
- GATT (Generic Attribute Profile) 지원
- MTU(Maximum Transmission Unit) 협상 지원

**기기 요구사항:**
- Bluetooth 칩셋: Qualcomm, Broadcom, MediaTek, Apple 등 주요 제조사
- GATT 최소 크기: 23 bytes (기본)
- MTU 확장 지원: 최대 517 bytes (권장)

**테스트 기기:**
- 스마트폰: Samsung Galaxy, LG, OnePlus, Google Pixel 등 다양한 제조사
- 태블릿: 주요 시장 기기 (최소 2개 제조사)
- 측정기 시뮬레이터 또는 실제 BLE 기기

**검증 방법:**
- AndroidManifest.xml: `<uses-feature android:name="android.hardware.bluetooth_le" />` 확인
- BLE 기능 확인: `BluetoothAdapter.isLe()` 또는 `getBluetoothLeAdvertiser()`
- nRF Connect로 GATT 호환성 검증
- 실제 기기에서 측정기 연결 테스트
