## 1. 개요

### 1.1 목적

본 문서는 WIMS 2.0 프로젝트의 전체 테스트 활동에 대한 총괄 계획을 수립한다. Phase 1(제품관리 시스템)과 Phase 2(견적설계, 발주관리, 제조관리, 현장실측)에 대해 단위 테스트부터 사용자 인수 테스트(UAT)까지의 테스트 전략, 환경, 일정, 합격 기준을 정의한다.

### 1.2 범위

| 구분 | 범위 |
|------|------|
| 대상 시스템 | WIMS 2.0 웹 애플리케이션 (5개 서브시스템) + Android 현장실측 앱 |
| 테스트 유형 | 단위 테스트, 통합 테스트, 시스템 테스트, 성능 테스트, 보안 테스트, UAT |
| Phase 1 범위 | ① 제품관리(PM) + 공통(CM) + MES REST API 연동 |
| Phase 2 범위 | ② 견적설계(ES) + ③ 발주관리(OM) + ④ 제조관리(MF) + ⑤ 현장실측(FS) |
| 제외 대상 | 현행 시스템(WIMS 1.0) 테스트, MES 내부 시스템 테스트 |

### 1.3 용어 정의

| 용어 | 정의 |
|------|------|
| 단위 테스트 | 개별 함수/메서드/컴포넌트 단위의 자동화 테스트 |
| 통합 테스트 | 모듈 간/시스템 간 연동을 검증하는 테스트 |
| E2E 테스트 | 사용자 시나리오 기반 전 과정(End-to-End) 검증 |
| UAT | 사용자 인수 테스트 — 현업 사용자가 실제 업무 시나리오로 검증 |
| 결함 | 요구사항 또는 기대 결과와 다른 시스템 동작 |
| 회귀 테스트 | 결함 수정 후 기존 기능에 영향 없음을 확인하는 재테스트 |

## 2. 테스트 전략

### 2.1 전략 개요

WIMS 2.0은 **Phase 분리 전략**을 적용한다. Phase 1과 Phase 2를 독립적으로 테스트하며, Phase 2 완료 후 전체 시스템 통합 테스트와 UAT를 수행한다.

```
Phase 1 (S3~S5)                    Phase 2 (S6~S15)
┌─────────────────────┐           ┌──────────────────────────────────┐
│ 단위 테스트 (S3~S4) │           │ 단위 테스트 (S6~S11)            │
│         ↓           │           │         ↓                       │
│ 통합 테스트 (S4)    │           │ 통합 테스트 (S12~S13)           │
│         ↓           │           │         ↓                       │
│ Gate 3: Phase 1 오픈│           │ 시스템 테스트 + 성능 (S13)      │
└─────────────────────┘           │         ↓                       │
                                  │ UAT (S14)                       │
                                  │         ↓                       │
                                  │ Gate 5: 전체 오픈               │
                                  └──────────────────────────────────┘
```

### 2.2 테스트 레벨별 전략

| 테스트 레벨 | 수행 주체 | 자동화 여부 | 대상 | 도구 |
|------------|----------|:-----------:|------|------|
| 단위 테스트 | 개발자 | 자동화 | 핵심 비즈니스 로직 | JUnit 5 (BE), Vitest (FE) |
| 통합 테스트 | 개발자 + QA | 반자동 | 모듈 간 연동, API 연동 | Spring Boot Test, Postman |
| 시스템 테스트 | QA | 수동 + 자동 | 전체 시스템 E2E | Playwright (웹), Espresso (앱) |
| 성능 테스트 | QA | 자동화 | 응답시간, 동시접속 | k6 또는 JMeter |
| 보안 테스트 | QA | 반자동 | 인증, 권한, 취약점 | OWASP ZAP |
| UAT | 현업 사용자 | 수동 | 실제 업무 시나리오 | 체크리스트 기반 |

### 2.3 테스트 우선순위

테스트 케이스 작성 및 실행 우선순위는 요구사항 우선순위와 연계한다.

| 우선순위 | 기준 | 테스트 비중 |
|---------|------|:-----------:|
| 최상 (P0) | 핵심 업무 기능, 데이터 정합성, 보안 | 100% 커버 |
| 상 (P1) | 주요 업무 기능, 성능 기준 충족 | 100% 커버 |
| 중 (P2) | 보조 기능, UX 개선 | 80% 이상 |
| 하 (P3) | 편의 기능, 부가 기능 | 주요 시나리오만 |

## 8. 결함 관리

### 8.1 결함 심각도 분류

| 심각도 | 정의 | 조치 기한 | 예시 |
|--------|------|----------|------|
| Critical | 시스템 중단, 데이터 손실/유출, 핵심 기능 불가 | 즉시 (24시간 이내) | BOM 데이터 삭제 오류, JWT 인증 우회, DB 데이터 손실, **frozen 스냅샷 변조 성공** |
| Major | 주요 기능 장애, 데이터 정합성 오류, 우회 불가 | 해당 스프린트 내 | 최적절단 결과 오류, 견적 금액 계산 오류, MES API 실패, **RuleEngine p95 SLA 지속 위반** |
| Minor | 부가 기능 오류, UX 불편, 우회 가능 | 다음 스프린트 내 | UI 정렬 깨짐, 에러 메시지 부적절, 도움말 오류 |
| Trivial | 외관 결함, 오타, 개선 제안 | 안정화 기간 | 색상 불일치, 문구 수정, 레이아웃 미세 조정 |

### 8.2 결함 관리 프로세스

```
발견 → 등록(GitHub Issues) → 분류(심각도/우선순위) → 배정 → 수정 → 검증 → 종료
                                     ↑                              │
                                     └── 재오픈 (검증 실패 시) ──────┘
```

| 단계 | 담당 | 도구 |
|------|------|------|
| 발견·등록 | QA / 테스트 참여자 | GitHub Issues (라벨: bug, severity) |
| 분류·배정 | PM (김지광) | GitHub Issues (담당자 지정) |
| 수정 | 개발자 | Git 브랜치 (bugfix/) |
| 검증 | QA (등록자) | 수정 확인 후 Issue Close |

### 8.3 결함 추적 라벨 체계

| 라벨 | 용도 |
|------|------|
| `severity:critical` | Critical 결함 |
| `severity:major` | Major 결함 |
| `severity:minor` | Minor 결함 |
| `phase:1` / `phase:2` | Phase 구분 |
| `subsystem:PM` / `ES` / `OM` / `MF` / `FS` / `CM` | 서브시스템 구분 |
| `area:rule-engine` / `area:decision-table` / `area:simulator` / `area:frozen` / `area:ast-cache` | **v1.1 신설** — BOM_RULE 관련 세부 영역 |
| `status:verified` | 수정 검증 완료 |
| `status:reopen` | 재오픈 |
