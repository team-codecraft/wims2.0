# DE41 초기 데이터 시드 입력 정책서 v1.0

**문서코드:** DE41
**버전:** v1.0
**작성일:** 2026-04-27 (S2 8일차)
**작성자:** 김지광 (PM, 코드크래프트)
**상태:** draft (Gate 2 진입 전 초판)

> [!abstract] 문서 개요
> 본 문서는 PP12-1 v2.2 §1 ●필수 산출물 **"데이터 전환 계획서 — 콘텐츠 재정의: 초기 데이터 시드 입력 정책"** 을 충족하며, **Gate 2 진입조건 "데이터 전환 매핑 정의 완료"** 를 만족하기 위한 단독 SOT 다.
>
> **핵심 정책 (PO 2026-04-27 결정):** WIMS 2.0 은 **빈 DB 로 시작**하며, 기존 As-Is 시스템 데이터의 전수 마이그레이션을 **실시하지 않는다**. As-Is 분석 결과(AN22-1 v1.1 §3·§4)는 **참고용**으로만 활용하고, **운영 시점 마스터 시드 데이터를 신규 입력**한다 (자재 87종 · 제품 25종 · 사용자 14명 · 코드 카탈로그 · 협력업체 11개사 등).
>
> **마이그레이션 색채 금지:** 본 문서는 "롤백 윈도우 / 전수 마이그레이션 / 병행 운영 SQL / 데이터 전수 이관 / 데이터 정제" 등 마이그레이션 활동을 전제하는 표현을 사용하지 않는다. 시드 입력은 사전 검증된 마스터 (BOM_26.04.24 보완본 · DE33-1 v1.2) 에서 가져오는 단방향 입력이다.
>
> **범위:** Phase 1 (PM 제품관리) 도메인 깊이로 작성. Phase 2 (ES·OM·MF·FS) 도메인은 [TBD] 마커로 자리확보.

## §1 개요

### 1.1 목적

본 문서는 다음 세 가지 목적을 갖는다.

1. **PP12-1 v2.2 §1 ●필수 산출물 충족** — 설계 단계 "데이터 전환 계획서 (콘텐츠 재정의: 초기 데이터 시드 입력 정책)". PP12-1 v2.2 §1 L58 표기 일관.
2. **Gate 2 진입조건 충족** — PP12-1 v2.2 §4 L122 진입조건 4건 중 "데이터 전환 매핑 정의 완료" 1건을 본 문서로 충족. 다른 3건 (BOM 논리/물리 ERD 확정 / MES API 규격서 확정 / 화면 설계서 확정) 은 별도 산출물 처리.
3. **마이그 미실시 · 빈 DB · 시드 입력 정책의 단독 SOT** — PO 2026-04-27 결정 사항을 구체적 실행 가이드로 변환한다. 도메인별 시드 규모 · 입력 도구 · 일정 · 검증 기준을 본 문서가 단독 SOT 로 정의한다.

> [!important] 마이그레이션 미실시 정책 (PO 2026-04-27 결정)
> WIMS 2.0 은 As-Is DB 로부터의 데이터 전수 이관을 실시하지 않는다. 결정 배경은 다음과 같다.
> 1. As-Is 데이터 품질 문제 (자재코드 중복 · 단가 이력 미관리 · 부자재 비활성. AN22-1 v1.1 §4.2 참조) 가 마이그 비용 대비 가치를 상회.
> 2. WIMS 2.0 은 BOM 계층 구조 · EBOM/MBOM 분리 · 옵션별규칙 등 **신설 데이터 모델** 비중이 커서 As-Is 평탄 매핑을 전환하는 비용이 신규 입력 비용을 초과.
> 3. 사용자 (TFT 14명) 가 신규 시스템 친숙화 단계에서 마스터 데이터 직접 입력을 통해 도메인 학습 효과 동시 달성.
>
> 본 문서는 위 결정 이후의 실행 가이드이며, 결정 자체의 재검토는 본 문서 범위 외다.

### 1.2 범위

본 문서는 빈 DB 로 시작하는 시점에 입력해야 할 마스터 시드 데이터의 정의 · 입력 방식 · 검증 기준을 다룬다.

| 영역 | Phase 1 (현재) | Phase 2 (후속) |
|---|---|---|
| 인증·사용자 시드 | ✅ 사용자 14명 (TFT) + 7 표준 ROLE (DE11-1 §7.3) | TFT 외 운영 사용자 점진 추가 (관리자 화면 운영 가이드) |
| 프로젝트 시드 | ✅ 0건 (운영 시점 신규 등록 — As-Is 미이관) | — |
| 자재·제품 시드 | ✅ 자재 87종 · 제품 25종 · 협력업체 11개사 (BOM_26.04.24 보완본 SOT) | 추가 자재 식별 시 점진 등록 |
| 시스템·공통 시드 | ✅ 코드 카탈로그 · 시스템 설정 12항목 | — |
| 견적·발주·도면·CAD | — (As-Is 미이관) | [TBD] §3.6 ~ §3.8 |

### 1.3 As-Is 참고 분류 (AN22-1 v1.1 §5.2 7개 도메인 요약)

본 문서는 As-Is 데이터를 **참고용으로만 분석**한 [[AN22-1_현행데이터분석서_v1.1]] §5.2 의 7개 도메인을 다음과 같이 시드 입력 정책으로 분류한다. 모든 도메인 공통: **As-Is 전수 마이그레이션 미실시**.

| # | 도메인 | As-Is 참고 (AN22-1) | 시드 입력 정책 (본 문서) |
|---|---|---|---|
| 1 | 인증·사용자 | 세션(Cookie) 기반 / SUPER·PR_MEM·ARCH_MEM·MEM 4역할 | 관리자 화면 신규 등록 (TFT 14명) + JWT 패스워드 신규 + 7 표준 ROLE |
| 2 | 프로젝트 | 기본 CRUD + 파일 로컬 저장 | 운영 시점 신규 등록 · 시드 0건 |
| 3 | 자재·제품 | 평탄 자재 목록 + 수동 제품 매핑 | 자재 87종 + 제품 25종 + 협력업체 11개사 일괄 시드 (Excel→CSV) |
| 4 | 견적설계 | 도면 기반 수동 견적 | As-Is 미이관 — Phase 2 진입 시 운영 시점 신규 등록 |
| 5 | 계약·발주 | 수동 발주서 + 단순 절단 계획 | As-Is 미이관 — Phase 2 진입 시 운영 시점 신규 등록 |
| 6 | 도면·CAD | 로컬 파일 저장 + 추정 메타데이터 | As-Is 미이관 — 처음부터 S3 적재 |
| 7 | 시스템·공통 | 추정 코드 카탈로그 + 시스템 설정 | 코드 카탈로그 + 시스템 설정 12항목 신규 시드 |

> [!note] As-Is 참고 활용 범위
> AN22-1 v1.1 §3 (도메인 분석) · §4 (현행 문제점) 는 본 문서 §3 도메인별 시드 명세 작성 시 "어떤 데이터가 필요한가" 의 후보 식별에만 활용된다. 시드 데이터의 실제 값은 BOM_26.04.24 보완본 · DE33-1 v1.2 등 사전 검증된 마스터에서 가져오며, As-Is DB 의 row 를 추출하지 않는다.

## §2 시드 입력 계획 (DE41-1 영역)

본 절은 **시드 입력의 전체 계획** 을 정의한다. 도메인별 상세 명세는 §3 에서 다룬다.

### 2.1 입력 전략

> [!info] Phase 분리
> Phase 1 (S5 D-Day 시점) 에 PM 도메인 일괄 시드를 완료한다. Phase 2 (ES·OM·MF·FS) 진입 시 도메인별 점진 입력으로 보강한다.

| Phase | 시점 | 대상 도메인 | 입력 방식 | 비고 |
|---|---|---|---|---|
| **Phase 1 일괄** | S5 D-Day (Phase 1 오픈) | 인증·사용자 / 자재·제품 / 시스템·공통 | 사전 준비된 시드 일괄 적재 | 본 §2~§3 의 핵심 |
| **Phase 1 점진** | S5 ~ S15 (운영) | 프로젝트 / 사용자 (TFT 외) / 추가 자재 | 화면 신규 등록 (점진) | 일상 운영의 일부 |
| Phase 2 일괄 | Phase 2 오픈 시점 | 견적·발주·도면 (시드가 필요하면) | [TBD — Phase 2 진입 시 본 문서 갱신] | §3.6~§3.8 자리확보 |

**Phase 1 일괄 vs 점진 구분 기준**: BOM 평가 (RESOLVED_BOM 산출) 가 동작하기 위해 **반드시 D-Day 이전** 에 적재되어야 하는 마스터는 일괄, 일상 업무 단위로 누적되는 트랜잭션성 데이터는 점진.

### 2.2 입력 일정

| 스프린트 | 기간 | 시드 관련 마일스톤 |
|---|---|---|
| **S2** (M2 1~2주차) | 2026-04-20 ~ 2026-05-03 | 시드 정의 (본 문서) · 시드 매트릭스 합의 · DE34-1 자재코드 체계 확정 |
| **S3** (M2 3주차) | 2026-05-04 ~ 2026-05-17 | 시드 데이터 준비 (Excel → CSV 변환 스크립트 1차) · Flyway 시드 V200 작성 |
| **S4** (M3) | 2026-05-18 ~ 2026-05-31 | 시드 데이터 검증 환경(STAGING) 적재 · 검증 SQL 카탈로그 작성 (§6.1) · 표본 정성 검증 |
| **S5** (M4 D-Day) | 2026-06-01 ~ 2026-06-14 | **Phase 1 일괄 시드 적재 (D-Day)** · 검증 결과 보고서 작성 · 운영 시작 |
| S6 ~ S15 | 2026-06-15 ~ 2026-11-01 | 점진 입력 운영 (사용자·프로젝트·추가 자재) · Phase 2 시드 정의 (S11~S12 예정) |

> [!note] 시드 데이터 동결 시점
> S4 종료 시점에 자재 87종 · 제품 25종 · 협력업체 11개사 · 사용자 14명 시드 데이터를 **동결** 한다. 이후 변경은 D-Day 이후 화면 신규 등록으로만 처리. 단, 시드 데이터 결함 발견 시 S5 D-Day 직전 핫픽스 예외 허용 (§2.6 리스크).

### 2.3 입력 도구

WIMS 2.0 은 시드 데이터 성격 별로 3가지 입력 도구를 사용한다.

| 도구 | 용도 | 적용 시드 |
|---|---|---|
| **Flyway 시드 마이그레이션** (`V200__seed_*.sql`) | 코드 카탈로그 · ROLE · 시스템 설정 등 정적 마스터 | code_catalog · role · permission · system_setting · option_group · option_value (DE33-1 §6 부분 활용) |
| **관리자 화면 신규 등록** | 사용자 · 권한 매핑 · JWT 패스워드 (보안 필요) | user 14명 (TFT) · user_role 매핑 |
| **Excel → CSV 일괄 적재** | 자재 · 제품 · 협력업체 · BOM 트리 · 다이스북 (대량) | item · product · item_supplier · dies_supplier · dies_book · dies_book_revision · standard_bom · ebom · mbom (대량) |

> [!info] Flyway 시드 vs 마이그 마이그레이션 (혼동 방지)
> 본 문서에서 사용하는 "Flyway 시드" 는 빈 테이블에 정적 마스터 row 를 INSERT 하는 행위 (`V200__seed_*.sql`) 다. As-Is DB 데이터를 옮기는 ETL 마이그레이션과 무관하다. DE33-1 §7.1 Flyway 파일 분리 전략 (V100~V106 스키마 / V200~ 시드) 을 따른다.

**Excel → CSV 변환 스크립트**: `scripts/seed/` 아래 Node.js 기반 변환 스크립트를 작성한다 (BOM_26.04.24 보완본 `.xlsx` → CSV → `LOAD DATA INFILE`). 단, MariaDB 권한 정책상 `LOAD DATA LOCAL INFILE` 로 적재. 스크립트 자체는 본 문서 범위 외 (S3 산출).

### 2.4 검증 정책

> [!important] 전수 검증 원칙
> Phase 1 시드 (자재 87종 · 제품 25종 · 사용자 14명 · 코드 카탈로그 전체) 는 **전수 검증** 한다. 시드 규모가 작아 표본 검증의 위험을 부담할 이유가 없다.

| 검증 단계 | 시점 | 도구 | 합격 기준 |
|---|---|---|---|
| 사전 검증 | S3~S4 STAGING | 검증 SQL (§6.1) + 자재코드 패턴 정규식 | 표준 패턴 위반 0건 |
| 적재 검증 | S5 D-Day | row count 비교 + FK 무결성 SQL | 계획 시드 규모 = 입력 후 row count |
| 사후 표본 정성 검증 | S5 D-Day +1일 | 자재 87종 중 10종 sampling 화면 표시 검증 | 화면에서 정상 표시 |
| 운영 시작 후 모니터링 | S5+ | 일상 모니터링 (관리자 화면) | — |

### 2.5 권한 / 책임

| 역할 | 책임 |
|---|---|
| **DBA** (코드크래프트) | Flyway 시드 SQL 작성 · STAGING 적재 · 적재 검증 SQL 실행 · 결과 보고 |
| **BE 리드** (코드크래프트) | 관리자 화면 신규 사용자 등록 기능 구현 (S3) · ROLE 매핑 검증 |
| **BA** (유니크시스템 협업) | 자재 87종 · 제품 25종 표본 정성 검증 (10종 sampling) · 운영 사용자 추가 인입 결정 |
| **PM** (코드크래프트) | 시드 일정 관리 · 동결 시점 통제 · 검증 결과 보고서 IM 단계 입력 |

### 3.2 인증·사용자 시드 (Phase 1)

#### 3.2.1 사용자 시드 (TFT 14명)

S5 D-Day 시점에 TFT 14명을 관리자 화면 신규 등록 한다. 화면을 통한 등록 이유는 (1) JWT 패스워드를 사용자별로 신규 설정해야 하고 (2) 등록 즉시 이메일 알림이 동작해야 하므로 Flyway INSERT 로 처리하기 부적합하기 때문이다.

| 컬럼 | 입력 출처 | 비고 |
|---|---|---|
| `username` | TFT 명단 (PM 보유) | 사번 또는 영문 ID |
| `email` | TFT 명단 | 회사 이메일 |
| `password_hash` | 신규 생성 (BCrypt) | 사용자별 초기 패스워드 발급 후 첫 로그인 강제 변경 |

| `org_role` | TFT 명단 | 직책 (PM/BA/BE/FE/QA/DBA/MES_OP) |
| `created_at` / `updated_at` | NOW() | 감사 컬럼 (DE33-1 §2.3) |

#### 3.2.3 user ↔ role 매핑

TFT 14명에 대한 ROLE 부여는 관리자 화면 등록 동시에 수행. 초기 매핑 가이드 (운영 정책):

- PM 직책 → `ROLE_PM_ADMIN`

> `ROLE_MES_READER` 는 사람 사용자가 아닌 MES 시스템 계정용 (토큰 기반). TFT 14명에 부여하지 않으며, S5 D-Day 시점에 별도 시스템 계정 1건을 생성한다.

## §4 시드 입력 표준 규칙

본 절은 §3 도메인별 시드 입력의 공통 표준 규칙을 정의한다. SOT 가 별도 문서인 항목은 인용으로만 처리하고 본문에 복사하지 않는다 (SOT 분열 차단).

### 4.1 자재코드 표준 패턴 적용

자재·제품 시드 적재 전 모든 자재코드는 [[DE34-1_표준데이터사전_v1.0]] §2 (자재 코드 체계 표준화 단독 SOT) 의 패턴과 일치해야 한다. 본 절은 검증 책임만 정의하며, 패턴 정의 자체는 DE34-1 §2 갱신.

**검증 단계** (S3~S4 STAGING 단계 사전 검증):

1. 정규식 일치 검증 — DE34-1 §2.1.1 ~ §2.1.4 의 5계열 패턴별 정규식 (예: 원자재 `^UNI-A\d{3}-\d{3}[A-Z]?$`)
2. ASCII 영숫자·하이픈만 허용 (DE34-1 §2.1)
3. 대소문자 정규화 (저장 시 대문자, DE34-1 §2.1)
4. itemCategory enum 일치 (DE34-1 §2.1.5)
5. is_phantom 강제 (반제품 4계열은 FALSE, DE34-1 §2.1.3)

> [!important] 본문 복사 금지
> 자재코드 패턴 정규식·매핑 규칙·금지어 등의 SOT 는 모두 [[DE34-1_표준데이터사전_v1.0]] §2 다. 본 §4.1 은 검증 책임만 정의하며 패턴 정의를 본 문서에 복사하지 않는다 (SOT 분열 차단). 신규 자재 식별 시 DE34-1 §2 갱신 → 본 §4.1 검증 SQL 갱신.

### 4.2 ROLE 매핑

[[DE11-1_소프트웨어_아키텍처_설계서_v1.2]] §7.3 역할 사전 (Role Dictionary) 단독 SOT 인용. 7 표준 ROLE (`ROLE_ADMIN` · `ROLE_PM_ADMIN` · `ROLE_PM_EDITOR` · `ROLE_PM_VIEWER` · `ROLE_RULE_EDITOR` · `ROLE_ESTIMATE_EDITOR` · `ROLE_MES_READER`) 정의는 DE11-1 §7.3 갱신. 시드 적재 책임은 §3.2.2.

> [!note] 레거시 ROLE 매핑
> AN22-1 v1.1 §3.8.3 의 4역할 (SUPER · PR_MEM · ARCH_MEM · MEM) 은 DE11-1 §7.3 마이그레이션 맵 참조. As-Is 사용자는 마이그하지 않으므로 시드 적재 시 표준 ROLE 만 사용.

### 4.3 인코딩·문자셋 (UTF-8)

| 영역 | 정책 |
|---|---|
| DB 문자셋 | `utf8mb4` · 콜레이션 `utf8mb4_general_ci` (DE33-1 §1.3) |
| CSV 파일 | UTF-8 (BOM 없음) |
| Excel → CSV 변환 시 | 변환 스크립트가 BOM 제거 후 저장 |
| Flyway 시드 SQL 파일 | UTF-8 |
| 응답 헤더 | `Content-Type: application/json; charset=UTF-8` (DE24-1) |

> [!warning] 한글 깨짐 사전 방지
> Excel UTF-8 저장 시 일부 환경에서 BOM (Byte Order Mark) 이 추가되어 `LOAD DATA INFILE` 첫 컬럼에 `﻿` 가 포함된다. 변환 스크립트가 BOM 제거를 명시적 처리. STAGING 적재 후 한글 row 표본 검증 (R1 대응).

## §6 검증 정책

### 6.1 시드 입력 검증 SQL 카탈로그 (Phase 1 깊이)

S5 D-Day 적재 직후 실행하는 검증 SQL 카탈로그. 모든 SQL 은 STAGING 환경에서 사전 실행하여 합격 기준 (§6.3) 을 만족함을 확인.

#### 6.1.1 row count 검증

```sql
-- 자재 87종
SELECT COUNT(*) FROM item WHERE deleted_at IS NULL;
-- 기대: 87 (원자재 40 + 부자재 47, 반제품 별도)

-- 제품 25종
SELECT COUNT(*) FROM product WHERE deleted_at IS NULL;
-- 기대: 25 (후렘 18 + 문짝 6 + 방충망 1)

-- 협력업체 11개사
SELECT COUNT(*) FROM dies_supplier WHERE deleted_at IS NULL;
-- 기대: 11

-- 사용자 14명 (TFT)
SELECT COUNT(*) FROM user WHERE deleted_at IS NULL;
-- 기대: 14 (+ MES 시스템 계정 1) = 15

-- 7 표준 ROLE
SELECT COUNT(*) FROM role;
-- 기대: 7
```

#### 6.1.2 자재코드 표준 패턴 검증 (DE34-1 §2.1 인용)

```sql
-- 원자재 패턴 위반 (UNI-A* / UNI-P* / YJN-* / DH-* / SC-* / ZZ-* 외)
SELECT item_code, item_name FROM item
WHERE bom_level = 'L5'
  AND item_code NOT REGEXP '^(UNI-[AP][0-9]{3}|YJN-|DH-|SC-|ZZ-)';
-- 기대: 0 row

-- 부자재 패턴 위반 (NN-XXXX 또는 X-001)
SELECT item_code, item_name FROM item
WHERE bom_level = 'L4'
  AND item_code NOT REGEXP '^([0-9]{2}-[0-9]{4}|X-001)$';
-- 기대: 0 row

-- 반제품 is_phantom=FALSE 강제
SELECT item_code FROM item
WHERE item_code REGEXP '^(F-|HC-|HX-|.*-H$)'
  AND is_phantom = TRUE;
-- 기대: 0 row (DE34-1 §2.1.3)
```

#### 6.1.3 FK 무결성 검증

```sql
-- item_supplier FK 위반 (item_id 또는 dies_supplier_id 가 존재하지 않음)
SELECT s.* FROM item_supplier s
LEFT JOIN item i ON s.item_id = i.id
LEFT JOIN dies_supplier d ON s.dies_supplier_id = d.id
WHERE i.id IS NULL OR d.id IS NULL;
-- 기대: 0 row

-- ebom 부모-자식 FK 위반
SELECT e.* FROM ebom e
LEFT JOIN standard_bom sb ON e.standard_bom_id = sb.id
WHERE sb.id IS NULL;
-- 기대: 0 row

-- bom_item_location 위치 코드 화이트리스트 검증 (10종)
SELECT * FROM bom_item_location
WHERE location_code NOT IN
  ('H01','H02','H02-1','H02-2','H03','H03-1','H03-2','W01','W02','W03');
-- 기대: 0 row (DE34-1 §2.1.4)
```

#### 6.1.4 코드 카탈로그 enum 누락 검증

```sql
-- itemCategory enum 누락
SELECT DISTINCT item_category FROM item
WHERE item_category IS NOT NULL
  AND item_category NOT IN
    ('PROFILE','GLASS','HARDWARE','CONSUMABLE','SEALANT','SCREEN');
-- 기대: 0 row

-- ROLE 표준 7종 외 사용 여부
SELECT DISTINCT role_code FROM user_role
WHERE role_code NOT IN
  ('ROLE_ADMIN','ROLE_PM_ADMIN','ROLE_PM_EDITOR','ROLE_PM_VIEWER',
   'ROLE_RULE_EDITOR','ROLE_ESTIMATE_EDITOR','ROLE_MES_READER');
-- 기대: 0 row (DE11-1 §7.3)
```

- 자재 추가 입력 — `SCR-PM-001 ~ 003` 자재관리 가이드 (자재코드 표준 패턴 자동 검증, DE34-1 §2.1)
- 협력업체 추가 — `SCR-PM-019 공급사 관리` 가이드
- 코드 카탈로그 추가 — `SCR-CM-006 코드관리` 가이드

### 7.3 As-Is 시스템 병행 운영 종료 시점

WIMS 2.0 D-Day (S5, 2026-06-01) 이후에도 As-Is 시스템은 일정 기간 참조용으로 유지될 수 있다. As-Is 종료 시점 결정은 본 문서 범위 외. 별도 운영 인수 산출물 (IM 단계 "운영 인수 계획서") 에서 확정.

> [!note] As-Is 병행 운영 정책 (PO 2026-04-27 결정)
> 마이그레이션 미실시 정책상 As-Is 와 WIMS 2.0 은 데이터 동기화 없이 독립 운영. As-Is 는 과거 프로젝트 조회 용도로만 유지하고, 신규 프로젝트는 WIMS 2.0 에서 시작. 따라서 데이터 동기화 SQL · 양방향 인터페이스 등 마이그 색채 활동을 추가하지 않는다.
